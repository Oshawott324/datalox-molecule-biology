# Hub -> Molecule Biology V1 Contract

Status: locked for V1 implementation planning.

This document defines the minimum contract between a Datalox hub/review surface
and the molecule-biology MCP server for the V1 trustworthy vertical. It does not
add biology scope. The diagnostic-digest workflow remains the current D1 script.

## Scope Boundary

In scope for V1:

- The hub launches or connects to the molecule-biology MCP server.
- The hub can run the diagnostic digest tool chain.
- The hub renders generated gel/map artifacts.
- The hub shows a replay/provenance bundle summary.
- The hub provides a final human review/approval frame.

Out of scope for V1:

- `edit_sequence`.
- Gibson, Golden Gate, Gateway, In-Fusion, or cloning wizards.
- Replacing the current D1 diagnostic digest design.
- Multi-user approval, auth, hosted execution, or database persistence.

The minimal local implementation in this repo is:

```bash
npm run demo:v1-review
```

That script runs the existing D1 MCP chain and writes a static review page:

```text
reports/review/v1-diagnostic-review.html
reports/review/v1-diagnostic-review.summary.json
```

It is a demo-critical review surface, not the final hub.

## HB1: Version Handshake

Problem: a hub cannot safely drive an MCP server unless it knows which tool
surface and envelope contract it is talking to.

Minimum V1 contract:

```ts
type MolBioVersionHandshake = {
  packageName: "@datalox/molecule-biology";
  packageVersion: string;
  protocol: "mcp-stdio";
  agentContractVersion: 1;
  toolCount: number;
  requiredTools: string[];
};
```

Required tools for the V1 diagnostic digest path:

```ts
[
  "open_sequence",
  "get_sequence_context",
  "simulate_digest",
  "find_restriction_sites",
  "render_digest_gel",
  "render_plasmid_map",
  "validate_workspace"
]
```

Acceptance:

- The hub calls `tools/list` and verifies every required tool exists before
  running D1.
- The hub rejects a missing required tool before the first domain call.
- The hub records package/protocol/tool-count metadata in the review summary.

Current local demo status:

- `scripts/demo-diagnostic-digest-mcp.mjs` uses real MCP stdio.
- `npm run smoke:mcp` verifies the stdio boundary and tool count.
- Full explicit hub handshake remains hub-side work.

## HB2: Provenance Bundle Schema

Problem: the hub must show evidence, not only final images.

V1 provenance source:

- Replay bundle generated by `packReplayBundle`.
- Bundle verified by `verifyReplayBundle`.
- The review surface displays bundle path, record count, and tool list.

Minimum review summary:

```ts
type MolBioReviewProvenance = {
  bundlePath: string;
  recordCount: number;
  tools: string[];
  workspacePath?: string; // scrub in shared/exported bundles
};
```

Acceptance:

- Review page must show replay status as verified.
- Review page must show record count.
- Review page must show the ordered tool chain or a compact equivalent.
- Shared bundles must apply the redaction policy in
  `docs/provenance-bundle-schema.md`.

Current local demo status:

- `npm run demo:v1-review` renders replay status, record count, bundle path, and
  tool chain.

## HB3: Hub Lifecycle and Transport Safety

Problem: the hub owns process lifecycle and cannot rely on a manually managed
terminal process.

Minimum V1 contract:

```ts
type MolBioServerLaunch = {
  command: "node";
  args: ["dist/src/cli/main.js", "mcp-server"];
  cwd: string;
  transport: "stdio";
  maxResponseBytes: 512000;
};
```

Acceptance:

- The hub starts the server with an explicit command, args, and cwd.
- The hub shuts down the child process after the review run.
- The hub enforces or respects a max response/frame size compatible with
  `RESPONSE_ENVELOPE_BYTE_CEILING`.
- The hub treats `ok:false` envelopes as domain/protocol failures, not as
  successful biology.

Current local demo status:

- `scripts/demo-diagnostic-digest-mcp.mjs` starts and closes a stdio MCP child.
- MB3 removed `structuredContent` duplication and added a response ceiling.
- Windows process-group cleanup remains final hub-side hardening.

## HB4: Review UI

Problem: correct files in a folder are not enough for a serious demo. The viewer
needs one frame that connects biology, artifacts, provenance, and human review.

Minimum V1 review frame:

- Diagnostic gel SVG.
- Forward and reverse plasmid map SVGs.
- Candidate enzyme table with the selected pair.
- Replay bundle status and record count.
- Ordered tool chain.
- Human review/approval panel.

Acceptance:

- One command generates a review page from a real MCP run.
- The review page embeds or renders the actual generated artifacts.
- The review page includes the replay/provenance summary.
- The review page does not alter the D1 scientific scenario.

Current local demo status:

```bash
npm run demo:v1-review
```

The output JSON prints:

```ts
{
  ok: true,
  reviewPath: string,
  summaryPath: string,
  workspaceDir: string,
  workspacePath: string,
  gelArtifact: ToolArtifact,
  mapArtifacts: Record<string, ToolArtifact>,
  bundlePath: string,
  recordCount: number
}
```

Open `reviewPath` in a browser for recording.

## Phase 1 Anchor

Phase 1 begins only after:

- MB6 is complete.
- HB1-HB4 are specified.
- The minimal HB4 review path is runnable.

Post-V1 work such as `edit_sequence`, `simulate_restriction_ligation` UX,
Gibson/Golden Gate, and richer SnapGene-style editing belongs after this gate.
